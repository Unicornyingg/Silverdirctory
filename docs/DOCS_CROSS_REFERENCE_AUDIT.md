## 🔴 Incorrect / Misleading

### 1. `proxy.ts` is dead code — not wired as middleware
**Doc ref:** TD §3 ("proxy.ts: admin key bootstrap for /admin/verify"), TD §6 ("proxy.ts strips valid ?key= and sets admin cookie for convenience")

**Reality:** `proxy.ts` exports a `proxy()` function and a `config` matcher
following the Next.js middleware convention, **but no `middleware.ts` file exists
in the repo root**, and no file imports `proxy.ts`. Next.js only auto-invokes
`middleware.ts` (or `middleware.js`) at the project root. Therefore:

- The admin `?key=` → cookie flow described in TD §6 and UG §6.1 **never
  executes** at runtime.
- Accessing `/admin/verify?key=<KEY>` will **not** set the
  `silver_admin_session` cookie automatically.
- The `/admin/verify` page must have its own internal auth check (it reads the
  cookie inside `page.tsx` via `cookies()`), but the cookie is never set.

**Fix options:**
1. Rename `proxy.ts` → `middleware.ts` so Next.js picks it up.
2. Or document the actual admin access pattern if auth is handled differently.

### 2. User Guide §4.1 — caregiver signup description omits "confirm password" field
**Doc ref:** UG §4.1 step 2: "Enter name, email, phone, and password."

**Reality:** `/for-nurses/page.tsx` collects **five** fields: full name, email,
phone, password, **and confirm password**. The form requires both and validates
they match. Omitting "confirm password" from the guide will confuse users who
see the extra field.

**Fix:** Change to "Enter name, email, phone, password, and confirm password."

---

### 3. User Guide §4.2 — caregiver login claims "email/password or phone OTP" but Google is also shown (client tab)
**Doc ref:** UG §4.2 step 2: "Use email/password or phone OTP flow."

**Reality in login page:** The caregiver tab shows SMS OTP **and** email/password
(correct). But the doc implies these are the only two methods. While Google
OAuth is intentionally blocked for caregivers (with an error message), the
description should note that Google login is **not available** for caregivers to
avoid confusion when users see the client tab offering it.

**Fix:** Add: "Google sign-in is only available for family accounts."

### 4. User Guide §3.2 — family login says "email/password or Google" but SMS OTP is also available
**Doc ref:** UG §3.2 step 2: "Use email/password or Google login."

**Reality:** The client tab in `/login` shows **three** methods: Google OAuth,
SMS OTP, and email/password. The SMS OTP flow for families is fully functional.

**Fix:** Change to "Use Google login, SMS code, or email/password."

### 5. Tech doc §9 — `public.users` described as having "moderation fields" — it's a VIEW not a table
**Doc ref:** TD §9: "public.users (view over auth.users + moderation fields)"

**Assessment:** The parenthetical "(view over auth.users + moderation fields)"
is actually correct! But the top-level heading "Main entities" lumps it with
tables, which is mildly misleading. The word "view" is present but easy to miss.
More importantly, `public.users` cannot be inserted into or updated directly —
this is relevant for anyone reading the data model section.

**Fix:** Change the bullet to:
`public.users` — **read-only view** joining `auth.users` with `user_moderation`

## 🟡 Inaccurate Detail

### 8. Tech doc §2 says "Resend (email helper utilities)" — no SDK, uses raw fetch
**Doc ref:** TD §2: "Resend (email helper utilities)"

**Reality:** There is no `resend` npm package in `package.json`. Both
`lib/email/send-verification-email.ts` and `lib/email/send-inquiry-email.ts` use
raw `fetch("https://api.resend.com/emails", ...)`. The description is not wrong
per se (Resend IS used), but readers may expect an SDK import. Stripe is
already described as "manual fetch integration" — Resend should match.

**Fix:** Change to: "Resend (email delivery via raw fetch to Resend API)"

---

### 9. `send-inquiry-email.ts` references "Golden Directory" — branding mismatch
**File:** `lib/email/send-inquiry-email.ts` lines 43, 54

**Reality:** The email body says "A family reached out through **Golden
Directory**" and signs off as "**Golden Directory**". The rest of the codebase
uses "Silver Directory" consistently. This is a branding bug, not a docs issue,
but it means the email feature described in docs would send incorrectly branded
emails.

### 10. `sendCaregiverInquiryEmail` is dead code — never imported
**File:** `lib/email/send-inquiry-email.ts`

**Reality:** `sendCaregiverInquiryEmail` is exported but has zero import
usages anywhere in the codebase. The `app/api/inquiries/` folder is empty, and
the legacy `inquiries` table is explicitly dropped in `schema.sql`. This utility
is orphaned — the feature was replaced by in-app chat.

The docs don't mention inquiry emails, which is correct, but the dead code
should be cleaned up.

### 11. Tech doc §7 omits `INQUIRY_FROM_EMAIL` env var used in code

### 12. Login page email/password is available for BOTH roles — doc implies it's caregiver-only
